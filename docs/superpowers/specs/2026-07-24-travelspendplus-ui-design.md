# TravelSpendPlus 界面设计方案（Plan B）

日期：2026-07-24
状态：已与用户逐块确认，待写实施计划
仓库：`TravelSpendPlus`

## 〇、背景与前置状态

Plan A（领域逻辑 + 本地持久化，无界面）已于 2026-07-23 完成并测试通过——`Money`、`Trip`、`Participant`、`Expense`、`BudgetCalculator`、`BalanceCalculator`、`CategoryBreakdownCalculator`、Drift 持久化 + `TripRepository`，全部在 `app/lib/domain/` 和 `app/lib/persistence/` 下，49 个测试通过。`app/lib/main.dart` 目前仍是 Flutter 新建项目自带的默认计数器模板，`app/lib/` 下没有任何界面代码，`pubspec.yaml` 也未引入图表库——本文档规划的界面部分（Plan B）是完全从零开始的下一步。

**重要澄清（2026-07-24）**：仓库里存在一份 2026-07-23 深夜写好但从未提交、从未实现的界面实施计划（`docs/superpowers/plans/2026-07-23-travelspendplus-ui.md`），范围包含完整的多人分摊/净余额界面(`BalancesScreen`)、英文硬编码文案、默认 Material 配色。经用户确认，这份计划与本文档的范围**不一致**（用户明确不需要分摊功能），已被删除，不作为参考。本文档是唯一有效的界面设计依据。

## 一、真实使用场景

主要使用者是用户的伴侣，单人（或两人共同、非多人分摊场景）记录旅行支出，核心诉求是"行程还没出发前几个月，就能把已经订好/预估的支出（机票、酒店等）先记进去，出发后再慢慢把预估改成实际"。示例：现在（2026-07）就要开始填"10月日本行"的计划中支出。

## 二、范围声明

**这一版做的**：
- 行程列表 + 新建/编辑行程
- 单趟行程详情：预算总览（总预算/计划中/已发生/预计还剩）、按币种查看总数的切换、类别饼图、支出列表
- 加/编辑支出，含"计划中→已发生"一键转换
- 行程汇率设置（维护该行程涉及的币种汇率表 + 修改行程本位币）
- 中文 + 英文 + 德文三语言界面（跟随系统语言，不做应用内语言切换开关）

**这一版明确不做**（留到以后需要多人使用时再加）：
- 参与人管理、"Paid by/Paid for"字段、净余额展示、结算标记 —— 即完整的分摊功能
- 联网自动查询实时汇率（汇率全部手动维护）
- 自定义/新增支出类别（沿用 TravelSpend 默认类别集）
- 支出照片附件、行程邀请/多设备同步

## 三、页面结构与导航

```
行程列表(首页)
  ├─ 卡片：行程名/日期/进度条/已花费或已计划-预算
  ├─ 右下角 ＋ → 新建行程
  └─ 点击某张卡 → 行程详情
       ├─ 顶部：行程名、日期
       ├─ 未出发：显示"距出发还有 X 天"
       ├─ 已出发：显示"每日剩余预算"（官方公式：剩余总预算 ÷ 剩余天数）
       ├─ 预算总览卡：总预算 / 计划中 / 已发生 / 预计还剩 + "切换查看币种"下拉
       ├─ 类别饼图 + 图例（≤6 类，遵循 TravelSpend 默认类别集）
       ├─ 支出列表：每笔显示类别/金额/计划中或已发生标签；点"计划中"记录可一键标记为已发生
       ├─ 右下角 ＋ → 加支出
       └─ 设置入口 → 编辑行程 / 汇率设置
             ├─ 编辑行程：名称/日期/总预算
             └─ 汇率设置：维护"币种=汇率"列表；修改本位币的操作也在这里
```

新建行程表单字段：名称、开始日期、结束日期、本位币、总预算。
加/编辑支出表单字段：金额+币种、类别（默认类别集单选）、备注、日期、状态（计划中/已发生）。

## 五、关键交互细节

- **切换查看币种**：预算总览卡上的下拉选择，选中后卡片数字按行程的汇率表实时换算显示；这是纯展示层面的切换，不修改行程本身设置，离开页面恢复默认（本位币）显示。
- **修改本位币（2026-07-24 修正）**：原方案"清空汇率表"有个严重问题——`Expense.amountInHomeCurrency` 和 `Trip.totalBudget` 都是按*当前*本位币存的具体数值，只清空汇率表并不会重算这些已经存好的数字，结果是本位币一改，所有历史支出和总预算的数字会被"贴上新币种标签但数值没变"，界面上悄悄显示错误的总额而不会报错。改成这样：修改本位币时，必须先填一个"1 [旧本位币] = ? [新本位币]"的换算率；有了这一个统一换算率，就能把 `totalBudget`、每一笔支出的 `amountInHomeCurrency`、以及汇率表里的每一条记录**按同一个比例统一重算**（因为它们原本全部统一用旧本位币计价），不需要清空重填，也不会丢数据。这是一个新的、原子性的仓储操作（`TripRepository` 需要新增对应方法），比"清空重填"更简单也更正确。
- **计划中→已发生**：支出列表里点"计划中"标签的记录，弹出确认框，可顺手把预估金额改成实际金额（不强制两者一致），确认后状态转为"已发生"。
- **未出发 vs 已出发**：行程开始日期(`startDate`)之前，预算总览卡显示"距出发还有 X 天"，不显示"每日剩余预算"；到达/超过开始日期后自动切换成官方"剩余总预算÷剩余天数"的动态每日预算。
- **三语言**：全部界面文案（标签、按钮、类别名、校验提示等）通过 Flutter `intl` + ARB 文件维护中/英/德三份翻译，跟随系统语言自动选择，不在应用内做语言切换按钮。
- **记账时遇到没有汇率的币种**：加支出表单里选择的币种如果不是本位币、且行程的汇率表里还没有这个币种的汇率，表单当场多出一个"1 [币种] = ? [本位币]"的输入框，一并填写后随这笔支出一起保存进行程的汇率表——不允许存在"选了外币却没有对应汇率、导致换算总数算不出来"的支出。
- **未出发 / 已出发 / 已结束 三种状态（2026-07-24 补充第三种）**：预算总览卡原方案只区分了"未出发"和"已出发"两种，遗漏了行程结束之后的状态——`BudgetCalculator.remainingDailyBudget` 在结束日期之后本来就返回 `null`（没有"剩余天数"这个概念了），所以详情页要按当前日期分三种情况显示：未出发(距出发还有X天)、进行中(每日剩余预算)、已结束(不显示每日预算，只显示静态总览，标注"行程已结束")。
- **切换查看币种的换算方向**：行程汇率表存的是"外币→本位币"（1外币=?本位币），但"切换查看币种"这个展示层面的功能需要反方向换算（本位币→其他币种，甚至外币→外币需要经本位币中转）——这不是直接复用现有 `ExchangeRate.convert()` 就能做到的，需要单独写一个双向换算的小函数，实施计划里要明确写出来，不能假设"复用现成的就行"。

## 六、对现有领域/持久化层的影响

Plan A 没有"行程级汇率表"这个概念（原设计文档只提到单次转换用的 `ExchangeRate` 工具类）。这一版界面需要新增：

- `Trip` 增加一个"该行程涉及币种→汇率"的列表（新领域字段，例如 `Map<String, double> exchangeRates`，key 是币种代码，value 是"1 单位该币种 = value 单位本位币"）
- Drift 数据库 schema 需要新增对应的表来持久化这份列表，`schemaVersion` 从 1 升到 2，写一个 Drift migration（目前没有真实设备数据，迁移逻辑可以很简单，但要显式写出来，不能默认省略）
- `TripRepository` 需要新增：
  - `updateTrip`：修改行程名称/日期/总预算（**目前完全不存在**，Plan A 只有 `createTrip`/`getTrip`，编辑行程功能没有这个方法就无法实现）
  - 读写汇率表的方法
  - `changeHomeCurrency(tripId, newCurrency, oldToNewRate)`：一个原子操作，按统一换算率同时重算 `totalBudget`、所有支出的 `amountInHomeCurrency`、汇率表里每一条记录（见第五节"修改本位币"的修正说明）

**类别改成稳定 key，不直接存显示文字**：`Expense.category`（Plan A 已经是裸 `String`）如果直接存"餐饮"这种显示文字，中/英/德三语言一切换，同一个类别就会因为存的文字不同被饼图/统计当成三个不同类别，统计直接碎掉。这一版起 `category` 存**稳定的英文 key**（`food`/`transport`/`lodging`/`shopping`/`entertainment`/`other`，固定 6 个，不可自定义，与前面"沿用默认类别集"的范围一致），显示时才通过三语言翻译表转成"餐饮/Food/Essen"这类文字——存储层永远只认 key，不认当前界面语言。

**`includeInSplit` 字段**：`Expense` 构造函数仍然要求这个字段（Plan A 已经写死），但这一版界面完全没有分摊功能、不会有任何界面读取这个值来做判断——直接固定传 `true`，不在界面上暴露开关。

这几项属于领域层/持久化层的新增工作，不是纯界面工作，会在实施计划里作为独立任务先做（复用 Plan A 已验证的 TDD 流程），界面任务在其基础上构建。

**读代码后发现的技术约束（2026-07-24 补充）**：Plan A 已经落地的 `Expense` 构造函数强制要求非空的 `paidBy`（单个 `Participant`）和非空的 `paidFor`（`List<Participant>`），`Trip` 也需要 `participants` 列表——这套约束是 Plan A 阶段就写死并测试过的，不是这一版可以绕开的东西。既然界面不做参与人管理，处理方式是：新建行程时在背后静默创建唯一一个默认 `Participant`（如 `name: '我'`），行程内所有支出的 `paidBy`/`paidFor` 都自动指向这同一个参与人，全程不出现在界面上、用户完全无感知。这不是新的产品决策，只是让"界面不做分摊"这个已经定好的范围，在现有领域层约束下能跑起来的必要实现细节。

## 七、空状态与校验

- 一个行程都没有：首页显示引导文案"还没有行程，点右下角开始规划你的第一趟旅行"
- 行程里还没有支出：详情页饼图/列表位置显示"还没有记账"引导语；预算总览正常显示（各项为 0）
- 表单校验（本地即时校验，不涉及联网）：
  - 金额必须 > 0
  - 结束日期不能早于开始日期
  - 名称/日期/金额/类别为必填，留空不允许保存

## 八、测试方法

- 领域层（预算计算、汇率换算、分类统计）已经测完，不重复测。
- 界面层用 Flutter widget test 覆盖关键场景：新建行程表单保存后出现在列表；加一笔支出后总览数字和饼图同步更新；标记"已发生"后计划中/已发生金额互相转移；表单校验拦截非法输入；未出发/已出发两种状态下预算总览卡的显示切换正确；修改本位币后汇率表被清空。
- 应用完全离线优先（汇率手动维护，不联网），本版本不涉及网络异常处理，主要错误处理面是表单校验和空状态。

## 九、明确排除（留到以后）

多人分摊完整功能（参与人管理、Paid by/Paid for、净余额、结算标记）、联网自动汇率、自定义类别、支出照片附件、行程邀请/多设备同步、深色模式、删除行程/删除支出（这一版不做任何删除操作，只有创建/编辑/标记已发生）。

"每日剩余预算"统一按 `includePlannedInDailyBudget: true` 计算（计划中支出计入每日预算），不在界面上做开关——`BudgetCalculator.remainingDailyBudget` 本身支持关闭这个参数，但这一版直接用默认值，不额外做切换界面。
